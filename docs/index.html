00 / WHAT SOLAR IS

One project model.
Established EDA tools.

Solar is a lightweight open hardware design platform and EDA workflow orchestrator for Linux. It validates one explicit project model, invokes established tools safely and organizes useful artifacts without replacing simulators or synthesizers.

Start a projectCreate a conventional Verilog project and run its first complete flow. Use the CLISee the stable project, build, artifact and diagnostic commands. Debug a simulationUnderstand named tests, VCD paths and when to run solar scan. Read internalsOpen the complete manifest, architecture, backend and testing references.
01 / INSTALL

One command.
No sudo.

The prebuilt release supports GNU/Linux x86_64 with glibc 2.35 or newer. The installer verifies SHA-256, installs below ~/.local, includes the private YANC 5.2 bundle and does not modify shell startup files.

curl -fsSL https://github.com/ART3121/solar/releases/latest/download/install.sh | sh

export PATH="$HOME/.local/bin:$PATH"
solar --version
solar doctor --all

External tools

Icarus Verilog and Yosys power the default Verilog flow and remain system packages. Verilator, cocotb, GTKWave and Surfer are optional. Solar never installs those tools automatically.

# Debian / Ubuntu
sudo apt install iverilog yosys verilator gtkwave

# Arch Linux
sudo pacman -S iverilog yosys verilator gtkwave

cocotb 2.x requires Verilator 5.036 or newer. Older Verilator packages still work for native Verilator simulations.

02 / QUICK START

From empty folder
to verified RTL.

mkdir counter
cd counter
solar init --template verilog
solar scan
solar check
solar build sim --list
solar build full
solar report

solar scan discovers conventional .v and .sv files below rtl/ and tb/, infers unambiguous tops and updates only managed manifest fields. Project loading also discovers conventional sources, so scan is not a prerequisite for every build.

Typical project tree

counter/
├── solar.toml
├── rtl/          authored RTL
├── tb/           authored testbenches
├── sim/          published VCD/FST
├── synth/        netlist and synthesis report
└── .solar/       private logs, state and workspaces
03 / PROJECT CONTRACT

The manifest says
what the flow means.

Paths in solar.toml are relative to the manifest. Format 2 supports named tests and profiles. Solar implements its documented TOML subset, not every TOML feature.

[solar]
format = 2

[project]
name = "counter"
language = "verilog"
default_test = "basic"
default_profile = "debug"

[sources]
rtl = ["rtl/counter.v"]

[simulation]
backend = "iverilog"

[synthesis]
backend = "yosys"
top = "counter"

[[profile]]
name = "debug"
defines = ["DEBUG"]

[[test]]
name = "basic"
top = "counter_tb"
sources = ["tb/counter_tb.v"]
waveform = "counter.vcd"

Use solar config set --name, --top and --test for supported transactional edits. Other values remain explicit manifest edits.

05 / SIMULATION AND WAVEFORMS

The testbench writes it.
Solar publishes it.

The waveform configured for a named test must agree with the literal path used by the testbench. Include the extension in both places:

// tb/counter_tb.v
initial begin
  $dumpfile("counter.vcd");
  $dumpvars(0, counter_tb);
end

# solar.toml
waveform = "counter.vcd"

If you change the literal $dumpfile path after a previous scan, run solar scan again so the managed test configuration is synchronized. Then rerun solar build sim basic.

A passing simulation may intentionally produce no waveform. Solar reports that clearly and does not invent a VCD. Use solar view basic only after a waveform has been registered.

06 / ARTIFACTS

Useful output visible.
Disposable state private.

  • sim/: validated Verilog VCD/FST waveforms.
  • synth/: Verilog netlist and synthesis report.
  • software/: generated CMM/C++ Assembly output.
  • hardware/: published SAPHO hardware.
  • simulation/: published SAPHO simulation output.
  • .solar/: private executables, logs, temporary files and state.

solar clean removes registered generated files only. It preserves source files, the manifest, unregistered files and legacy layouts.

07 / TROUBLESHOOTING

Start with the
first useful diagnostic.

Tool not found

Run solar doctor --all, install the missing external package through your distribution and retry. Solar does not download tools.

Wrong or missing waveform

Compare the test's waveform value with $dumpfile, including .vcd or .fst. Run solar scan after changing the literal testbench path.

Wrong simulation selected

Use solar build sim --list. Select a name explicitly or set a valid default with solar config set --test NAME.

Synthesis includes a testbench

Keep authored design files in rtl/ and testbenches in tb/. Solar synthesis uses RTL and the selected profile, never test-specific sources.

Need the last failure

Run solar report. It can display a persisted failed build and does not execute another build.

o significado do fluxo.

[[test]]

05 / SIMULAÇÃO E FORMAS DE ONDA

O testbench grava.
O Solar publica.

A forma de onda configurada no teste deve coincidir com o caminho literal usado pelo testbench. Inclua a extensão nos dois lugares:

// tb/counter_tb.v
initial begin
  $dumpfile("counter.vcd");
  $dumpvars(0, counter_tb);
end

# solar.toml
waveform = "counter.vcd"

Se você alterar o caminho literal de $dumpfile depois de um scan, execute solar scan novamente para sincronizar a configuração gerenciada. Depois rode solar build sim basic.

Uma simulação aprovada pode não produzir forma de onda intencionalmente. O Solar informa isso e não inventa um VCD. Use solar view basic somente quando houver uma onda registrada.

07 / SOLUÇÃO DE PROBLEMAS

Comece pelo primeiro
diagnóstico útil.

Ferramenta não encontrada

Execute solar doctor --all, instale o pacote externo pela distribuição e tente novamente. O Solar não baixa ferramentas.

Forma de onda errada ou ausente

Compare o valor waveform do teste com $dumpfile, incluindo .vcd ou .fst. Execute solar scan após alterar o caminho literal do testbench.

Simulação errada selecionada

Use solar build sim --list. Selecione um nome ou defina o padrão com solar config set --test NOME.

Precisa ver a última falha

Execute solar report. Ele exibe um build com falha já registrado sem executar outro build.
